# 数据库自动分区

# partition 的一些限制

  1. 分区所涉及的列必须是每一个主键和唯一键的子集

  2. 关于InnoDB引擎的限制 InnoDB存储引擎中，分区和外键不兼容，不能同时使用。 ALTER TABLE ... OPTIMIZE PARTITION语句不能正常使用，用ALTER TABLE ... REBUILD PARTITION and ALTER TABLE ... ANALYZE PARTITION代替 不要给任何分区或任何子分区指定存储引擎，因为同一个表的所有分区的存储引擎必须一致。

  3. 关于一些sql函数的限制
    只有以下函数能够在分区表达式中使用（注意不是mysql语句，是创建表时的分区表达式） ABS()

    CEILING() (see CEILING() and FLOOR())

    DATEDIFF()

    DAY()

    DAYOFMONTH()

    DAYOFWEEK()

    DAYOFYEAR()

    EXTRACT() (see EXTRACT() function with WEEK specifier)：有个BUG：指定单元为 WEEK 时，不允许将其作为分区函数

    FLOOR() (see CEILING() and FLOOR())

    HOUR()

    MICROSECOND()

    MINUTE()

    MOD()

    MONTH()

    QUARTER()

    SECOND()

    TIME_TO_SEC()

    TO_DAYS()

    TO_SECONDS()

    UNIX_TIMESTAMP() (with TIMESTAMP columns)

    WEEKDAY()

    YEAR()

    YEARWEEK()

  4. 关于锁的限制 select 只锁包含要查询的数据的分区 update 只有当没有分区列被更新的时候才不用全表锁 insert 只有要被新增的分区加锁。但如果设置了自增列，则全表全分区锁

CREATE TABLE employees (
    id INT NOT NULL,
    fname VARCHAR(30),
    lname VARCHAR(30),
    hired DATE NOT NULL DEFAULT '1970-01-01',
    separated DATE NOT NULL DEFAULT '9999-12-31',
    job_code INT NOT NULL,
    store_id INT NOT NULL
)
PARTITION BY RANGE (store_id) (
    PARTITION p0 VALUES LESS THAN (6),
    PARTITION p1 VALUES LESS THAN (11),
    PARTITION p2 VALUES LESS THAN (16),
    PARTITION p3 VALUES LESS THAN (21)
);

通过sql语句来创建带有分区选项的表。

创建分区的提示

分区的定义需要从低到高来排序，不可以打乱顺序。

提高查询速度的建议

where查询条件最好包含分区列 并使用=,<, >, <=, >=, and <>或者IN () 表达式，这样查询的时候mysql会自动忽略不包含目标数据的分区，只查询包含目标数据的分区，从而提高查询速度

注意null

在list分区中，插入null数据会导致报错， 必须显式定义一个包含null的list分区

在hash 和range中，null被当作是最小的 存放在第一个分区

# 使用分区遇到的一些问题

数据库表使用了主键，但是由于主键必须包含分区列。此时可以删除原有主键，创建一个新的联合主键，新的主键不能是null；

ALTER TABLE <table>
add primary key(<column1>,<column2>);

# 自动定时分区

show variables like '%event_scheduler';
# 开启时间定时计划
set global event_scheduler=1;
ALTER TABLE  bn_load_profile  PARTITION BY RANGE (to_days(value_time)) (
        PARTITION blp_his  VALUES LESS THAN (to_days('20230903')+1),
        PARTITION blp_20240630  VALUES LESS THAN (to_days('20240630')+1),
				
        PARTITION blp_other  VALUES LESS THAN MAXVALUE );
ALTER TABLE bn_load_profile
REORGANIZE PARTITION blp_other INTO (
				PARTITION blp_20240701  VALUES LESS THAN (to_days('20240701')+1),
        PARTITION blp_20240731  VALUES LESS THAN (to_days('20240731')+1),
				
				PARTITION blp_other  VALUES LESS THAN MAXVALUE );
